V8 垃圾回收 机制

V8 是 Google 开发的高性能 JavaScript 引擎，其垃圾回收（GC）系统经过多年的优化， 采用了分代收集、并发标记、增量清除等先进技术，让 JavaScript 开发者无需手动管理内存。

💡 为什么需要 GC

自动内存管理的必要性

JavaScript 是一门自动内存管理的语言。V8 的垃圾回收器自动识别并回收不再使用的对象， 防止内存泄漏，让开发者专注于业务逻辑。

🔒

防止内存泄漏

自动识别不再被引用的对象并释放内存，避免程序因内存耗尽而崩溃。

⚡

低延迟保证

通过并发标记和增量清除，将 GC 暂停时间控制在毫秒级，保证用户体验。

🧠

智能分代策略

基于「大多数对象生命周期极短」的经验法则，对新老对象采用不同的回收策略。

🔄

并发与并行

充分利用多核 CPU，将 GC 工作分散到多个线程，减少主线程阻塞。

📐

内存压缩整理

定期整理内存碎片，提高内存分配效率和缓存命中率。

🎯

Orinoco 优化

V8 的现代 GC 代号 Orinoco，包含多种先进优化技术持续迭代。

📦 内存分代

分代假说与内存布局

V8 将堆内存分为不同的空间，基于分代假说（Generational Hypothesis）： 大多数对象的生命周期极短，而存活较久的对象往往继续存活。

🔄 新生代回收

Scavenge 算法（Cheney 算法）

新生代采用 Cheney 的半区复制算法。将新生代空间分为两个等大的半区：From Space 和 To Space， 交替使用。

From Space（使用中）
🟢 obj_a (存活)
⚪ obj_b (死)
🟢 obj_c (存活)
⚪ obj_d (死)
🟢 obj_e (存活)
➡️
To Space（空闲）
obj_a (复制)
obj_c (复制)
obj_e (复制)

1
Scavenge 算法执行步骤
1

检查触发条件

当 From Space 的内存使用量达到一定阈值时，触发 Scavenge。新对象优先分配在 From Space。

2

根对象遍历（Root Set Tracing）

从 GC Roots（栈、全局对象、寄存器中的引用等）出发，递归遍历所有可达对象，标记为「存活」。

3

复制存活对象到 To Space

将 From Space 中所有存活的对象按顺序复制到 To Space，实现内存紧凑排列（无碎片）。对象地址更新。

4

角色互换

From Space 和 To Space 角色互换。原来的 To Space 变为新的 From Space。原 From Space 整体清空，变为新的 To Space。

5

晋升判断

如果一个对象已经经历过一次 Scavenge 且仍然存活，或者 To Space 的使用率超过 25%，则将该对象晋升（Promote）到老生代。

⚠️ 为什么 25% 是晋升阈值？ 当 To Space 使用率超过 25% 时，剩余空间不足以保证下一次分配的高效进行，同时也说明该对象可能长期存活。及时晋升到老生代可以避免反复在新生代中复制这些对象，节省 CPU 开销。
特性 Scavenge（新生代） Mark-Sweep-Compact（老生代）
空间利用 始终浪费 50%（半区空闲） 充分利用整个空间
时间复杂度 O(存活对象数) — 与存活量成正比 O(堆大小) — 需遍历整个堆
内存碎片 无碎片（紧凑复制） Mark-Sweep 有碎片，Compact 解决碎片
适用场景 对象存活率低（<5%） 对象存活率高、空间大
暂停时间 极短（<1ms 通常） 较长（需要并发/增量优化）
🏗️ 老生代回收

Mark-Sweep-Compact 算法

老生代采用标记-清除-整理（Mark-Sweep-Compact）算法，这是最经典的垃圾回收策略之一。 V8 将其拆分为三个阶段并引入并发/增量优化。

🏷️ 标记阶段（Mark）— 识别存活对象

从 GC Roots 出发，递归遍历所有可达对象并标记为「存活」。V8 采用三色标记法：
• 白色 — 未访问的对象（可能是垃圾）
• 灰色 — 已发现但子对象未完全遍历
• 黑色 — 已完全遍历，确认存活

并发标记 Concurrent 增量标记 Incremental

✂️ 清除阶段（Sweep）— 回收死对象

遍历整个老生代内存，将所有未被标记的对象（白色）回收，释放其占用的内存空间。 V8 使用空闲链表（Free List）来管理回收后的内存块，方便后续快速分配。

增量清除 Incremental

🧩 整理阶段（Compact）— 内存碎片整理

当内存碎片率达到一定阈值时，V8 会触发整理阶段。将存活对象移动到一起， 消除内存中的空洞，形成连续的内存空间。这是最耗时的阶段。

并行整理 Parallel

📌 指针更新阶段（Update Pointers）

Compact 阶段移动对象后，所有指向被移动对象的指针都需要更新为新地址。 V8 使用前进指针（Forwarding Address）机制来高效完成指针修正。

gc-example.js 
// 标记阶段的简化实现逻辑（概念示意）
function mark(root) {
    const stack = [root];

    while (stack.length > 0) {
        const obj = stack.pop();

        if (obj.color !== 'black') {
            obj.color = 'black'; // 标记为存活

            // 遍历所有引用的子对象
            for (const ref of obj.references) {
                if (ref.color !== 'black') {
                    ref.color = 'gray'; // 标记为待处理
                    stack.push(ref);
                }
            }
        }
    }
}

// 清除阶段
function sweep(heap) {
    for (const obj of heap) {
        if (obj.color === 'white') {
            free(obj); // 回收未标记的对象
        } else {
            obj.color = 'white'; // 重置为下一轮准备
        }
    }
}
🚀 Orinoco 优化

减少 GC 停顿的先进策略

Orinoco 是 V8 垃圾回收器的代号。它采用多种策略来最小化 GC 对应用的暂停时间（Stop-The-World）。

🧵

并发标记 (Concurrent Marking)

标记工作在后台辅助线程上执行，不阻塞主线程的 JavaScript 执行。这是最关键的优化，将最耗时的标记阶段从暂停中移除。

最核心优化
⏱️

增量标记 (Incremental Marking)

将标记工作拆分为小步骤，穿插在 JavaScript 执行之间。每次只标记一小部分，避免长时间暂停。

~1ms 每步
🧹

并发清除 (Concurrent Sweeping)

清除阶段也在后台线程并发执行。主线程只做少量的簿记工作，大大减少了 STW 暂停时间。

并行处理
📦

并行整理 (Parallel Compaction)

内存碎片整理阶段使用多个辅助线程并行执行，加速老生代的内存整理过程。

多线程
🔀

写屏障 (Write Barrier)

并发标记期间，JS 代码可能修改对象引用。写屏障通过记录引用变更，确保并发标记的正确性（维护三色不变式）。

slot->value = new_val
📊

空闲时 GC (Idle-time GC)

利用浏览器的空闲时间（requestIdleCallback）执行 GC 工作，在用户无感知的情况下清理垃圾。

requestIdleCallback()
🔬
写屏障深度解析

并发标记面临的核心挑战是：当后台线程在标记时，主线程的 JavaScript 可能修改对象引用关系。 例如：一个黑色对象突然引用了一个白色对象（应该是垃圾），如果没有检测机制，这个白色对象会被错误回收。

write-barrier-example.js 
// 危险场景：黑色对象引用白色对象

const parent = { name: 'parent' };  // 已标记为黑色 ✅
const child  = { name: 'child'  };  // 未标记 (白色) ⬜

// 并发标记期间，主线程执行：
parent.ref = child;
// ⚠️ 写屏障触发！
// 检测到黑色对象(parent)引用了白色对象(child)
// 将 child 标记为灰色，加入标记队列
// → child 不会被错误回收 ✅

🐛 内存泄漏

常见的内存泄漏场景

即使有自动 GC，不正确的编码习惯仍然会导致内存泄漏。以下是 JavaScript 中最常见的内存泄漏场景。

👆 意外的全局变量

在函数中忘记使用 let/const，导致变量被提升为全局变量，永远不会被回收。

function leak() {
    // 缺少 let/const，变成全局变量
    hugeArray = new Array(1000000);
}

🔒 闭包引用

闭包持有的外部变量引用不会被释放，如果闭包本身被长期持有，其引用的整个作用域都不会被回收。

function createHandler() {
    const hugeData = new Array(1e6);
    return () => hugeData.length;
    // hugeData 被闭包引用，无法回收
}

📡 未清理的事件监听器

给 DOM 元素添加了事件监听器但在元素移除时忘记移除监听器，导致关联的对象无法被 GC 回收。

// ❌ 错误：移除元素但不移除监听器
element.addEventListener('click', handler);
element.remove(); // 监听器仍在内存中！

// ✅ 正确
element.removeEventListener('click', handler);
element.remove();

🗂️ 被遗忘的定时器

setInterval / setTimeout 回调中引用了外部大对象，未在适当时机调用 clearInterval / clearTimeout。

const timer = setInterval(() => {
    // 引用了 hugeObject，无法被回收
    doSomething(hugeObject);
}, 1000);
// 忘记 clearInterval(timer) 😱

💾 缓存无限增长

使用对象作为缓存 Map 但没有清理策略（如 LRU），导致缓存无限增长。

// ❌ 无限制缓存
const cache = {};
function getData(key) {
    if (!cache[key]) {
        cache[key] = fetchHugeData(key);
    }
    return cache[key];
}
// ✅ 使用 WeakMap 或限制大小

🔗 DOM 引用未清除

在 JavaScript 对象中保存了对已移除 DOM 元素的引用，导致 DOM 元素和关联数据都无法被回收。

const elements = {};
elements.btn = document.getElementById('btn');
// 后来 DOM 中移除了 #btn
// 但 elements.btn 仍持有引用！
// ✅ 用完置 null
🛠️ 现代 API

帮助 GC 的现代 JavaScript API

ES2021 引入了 WeakRef 和 FinalizationRegistry，让开发者可以更精细地参与垃圾回收过程。

🔗

WeakRef

弱引用。不会阻止 GC 回收被引用的对象。通过 deref() 访问，可能返回 undefined。

let obj = { data: 'hello' };
const ref = new WeakRef(obj);

obj = null; // 可被GC
ref.deref(); // undefined or {data}
📋

FinalizationRegistry

在对象被 GC 回收后执行回调。用于清理关联资源（如关闭文件句柄、网络连接）。

const reg = new FinalizationRegistry(
    (heldValue) => {
        console.log(heldValue + ' 被回收');
    }
);
reg.register(obj, 'myObj');
📦

WeakMap / WeakSet

键为弱引用的 Map/Set。当键对象无其他引用时，对应条目自动被 GC 清除。

const wm = new WeakMap();
let key = {};
wm.set(key, 'data');

key = null; // 条目自动清除
🔍 调试工具

如何检测内存问题

掌握工具是排查内存泄漏的关键。以下是最常用的内存调试工具和技巧。

🖼️

Chrome DevTools - Memory

Heap Snapshot：拍堆快照对比内存变化。
Allocation Timeline：实时观察内存分配。
Allocation Sampling：低开销的分配分析。

📊

Performance Monitor

实时监控 JS Heap 大小、DOM 节点数、事件监听器数量。锯齿形是正常的，持续上升则有泄漏。

🖥️

Node.js --inspect

使用 --inspect 标志启动 Node.js，连接 Chrome DevTools 进行内存分析。
--expose-gc 手动触发 GC。

💡 排查内存泄漏三步法 1. 拍堆快照：在可疑操作前各拍一次 Heap Snapshot。
2. 对比快照：比较两次快照中「分配但未释放」的对象，找到泄漏对象。
3. 追踪引用链：在 Retainers 面板查看泄漏对象被谁引用，定位代码根因。

最佳实践

• 使用 let/const 避免全局变量
• 及时移除不再需要的事件监听器
• 用 WeakMap 替代对象做缓存
• 组件销毁时清理定时器
• 避免在循环中创建大量闭包
• 使用 WeakRef 管理大对象引用
